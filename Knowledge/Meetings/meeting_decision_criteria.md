# KIBA 회의 의사결정 기준 v1

> 목적: 일일 회의에서 나온 결론을 같은 기준으로 검토하고, 근거가 부족한 결정을 실행 전에 멈춰 세운다.
> 사용법: 회의록과 함께 NotebookLM 소스로 등록한 뒤 `meetings/NOTEBOOKLM_VALIDATION_PROMPT.md`의 질문을 사용한다.

## 판정 등급

- `통과`: 근거와 담당·기한이 충분해 바로 실행할 수 있다.
- `조건부`: 실행 전 보완할 조건이 명확하다.
- `보류`: 법적·보안·비용·품질 위험 또는 핵심 근거 부족으로 실행하면 안 된다.
- `해당 없음`: 이번 결정과 관계없는 기준이다.

## 25개 기준

| ID | 판단 기준 | 회의에서 확인할 질문 | 통과 근거 |

| OBJ-02 | 우선순위 | 지금 해야 하는 이유와 미뤘을 때의 영향이 설명됐는가? | 긴급도·중요도 근거가 있음 |
| OBJ-03 | 범위 | 포함·제외 범위와 완료 기준이 명확한가? | 산출물·완료 조건이 있음 |
| OBJ-04 | 사용자 가치 | 실제 사용자 또는 업무 담당자의 불편을 줄이는가? | 대상 사용자와 개선 효과가 있음 |

| VAL-02 | 반복 절감 | 반복 업무 시간이나 오류를 측정 가능하게 줄이는가? | 현재·목표 지표가 있음 |
| VAL-03 | 유지보수성 | 담당자가 바뀌어도 운영 가능한 구조인가? | 문서·자동화·소유자가 있음 |
| VAL-04 | 확장성 | 다른 사업·양식·팀에도 재사용할 여지가 있는가? | 변수화·템플릿화 계획이 있음 |
| VAL-05 | 종료 가능성 | 효과가 없을 때 안전하게 중단·되돌릴 수 있는가? | 롤백·중단 기준이 있음 |
| RSK-01 | 법규·계약 | 관련 법령, 협회 기준, 계약 조건을 확인했는가? | 적용 기준과 출처가 있음 |
| RSK-02 | 개인정보 | 개인정보의 수집·저장·공개 범위가 최소화됐는가? | 비공개 저장·마스킹 기준이 있음 |
| RSK-03 | 영업기밀 | 원가·인력·고객 자료의 공개 범위가 통제되는가? | 접근 권한과 저장 위치가 있음 |
| RSK-04 | 자격증명 | API 키·비밀번호·토큰이 코드나 공개 문서에 노출되지 않는가? | Secret/DPAPI 등 보호 방식이 있음 |

| EXE-01 | 담당자 | 실행 책임자가 한 명 이상 지정됐는가? | `@담당자`가 기록됨 |
| EXE-02 | 기한 | 완료 또는 재검토 날짜가 지정됐는가? | `~YYYY-MM-DD`가 기록됨 |
| EXE-03 | 선행조건 | 권한·데이터·승인 등 시작 전 조건이 확인됐는가? | 의존성과 확보 상태가 있음 |
| EXE-04 | 작업 분해 | 하루 이상 걸리는 일은 검증 가능한 단계로 나뉘었는가? | 중간 산출물과 순서가 있음 |
| EXE-05 | 추적 연결 | 결정과 할 일이 GitHub Issue/Todo에 연결됐는가? | 이슈 번호 또는 Todo 경로가 있음 |
| QUA-01 | 입력 검증 | 입력 자료의 출처·버전·필수값을 확인했는가? | 검증 규칙과 오류 처리가 있음 |
| QUA-02 | 결과 검증 | 정답 샘플·대조값·사람 검토 절차가 있는가? | 테스트 또는 검수자가 있음 |
| QUA-03 | 기록성 | 결정 이유와 변경 이력이 나중에 추적 가능한가? | 회의록·커밋·이슈 기록이 있음 |
| QUA-04 | 전달성 | 비개발자도 결과와 다음 행동을 이해할 수 있는가? | 쉬운 요약과 담당·기한이 있음 |
| QUA-05 | 회고 | 실행 후 효과·오류·다음 개선을 확인할 시점이 있는가? | 회고 날짜 또는 측정 지표가 있음 |

## 최소 통과 규칙

1. `RSK-*`에서 하나라도 `보류`면 실행하지 않는다.
2. `EXE-01`, `EXE-02`, `EXE-05`가 빠진 할 일은 접수만 하고 실행 대기 상태로 둔다.
3. `OBJ-01`, `OBJ-03`, `QUA-02` 중 두 개 이상이 `조건부`면 다음 회의에서 다시 결정한다.
4. NotebookLM 답변은 근거 문장을 함께 제시해야 하며, 근거 없는 판정은 사람이 재검토한다.
